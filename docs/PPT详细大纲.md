### 具体内容

**场景引入:**
> "3个朋友分别在北大、清华、国贸，想一起喝咖啡，去哪？"

**痛点 (左侧红色):**
- 手动协调平均耗时 **15-30分钟**
- 结果往往不公平 - 总有人跑最远
- 现有地图只能搜"我附近"
- 无法计算"我们中间"

**机会 (右侧绿色):**
- **市场空白**: 无专注"多人公平聚会"的产品
- 高德/百度地图无多人中点功能
- 年轻人社交需求旺盛
- 企业团建/客户拜访是刚需

**金句 (底部大字):**
> "不是找最近的地方，而是找最公平的地方"

### 演讲话术
> "设想一个场景：3个朋友分别在北大、清华、国贸，想一起喝咖啡。
>
> 群里开始讨论：你说三里屯，我说太远；他说朝阳大悦城，她说没好店...
>
> 15-30分钟过去了，还没定下来。最后总有人妥协，打车40分钟去最远的地方。
>
> 核心问题是：**现有工具都在解决'去哪吃'，没人解决'对所有人都公平地去哪吃'**。"

### 三大技术亮点

| 亮点 | 技术实现 | 效果 |
|------|----------|------|
| **球面几何中点** | Haversine公式<br>2人球面中点 / 3+人加权质心 | 比平面算法更精确<br>跨城市误差可达数十公里 |
| **异步并发搜索** | Python asyncio<br>多场景同时请求 | 响应时间<1秒<br>支持多场景组合 |
| **智能地址识别** | 60+高校简称映射<br>地标名称识别 | "北大"→"北京大学"<br>避免地理歧义 |

### 技术栈
```
FastAPI + Amap API + Python Async + Jinja2
```

### 演讲话术
> "技术架构很清晰：用户输入 → 地址解析 → 中点计算 → POI搜索 → 智能排序 → 结果生成。
>
> 三个技术亮点：
>
> 第一，**球面几何中点**。不是简单的经纬度平均，是用Haversine公式计算地球曲面上的真实中点。
>
> 第二，**异步并发搜索**。多个场景同时请求，响应时间控制在1秒内。
>
> 第三，**智能地址识别**。内置60+高校简称，'北大'自动识别为北京大学，避免歧义。"
